# OSINT 开源情报分析技能

> 本技能整合了国际标准OSINT情报方法论（基于Cambridge、IEEE、Springer等学术研究）和工具链，帮助用户高效完成商业情报收集与分析任务。

## 一、任务目标

本技能用于：
- **商业竞争情报分析**：调研竞争对手的产品、市场动态、技术布局
- **友商追踪**：监控目标企业的最新动态、产品发布、人事变动
- **市场调研**：收集行业趋势、市场规模、竞品信息
- **背景调查**：人员背景核实、企业资信评估、招聘背调
- **信息验证**：验证新闻、图片、文档的真实性、事实核查
- **风险预警**：发现潜在风险信号（法律诉讼、监管处罚、舆论危机）
- **地缘政治分析**：国际事件追踪、冲突分析、政策影响评估
- **学术情报追踪**：研究领域动态、学者网络、技术趋势
- **舆情监控**：品牌舆情、危机预警、传播分析
- **供应链情报**：供应商分析、风险评估、替代方案
- **网络安全威胁情报**：攻击溯源、APT组织关联、IOC分析
- **反欺诈调查**：保险欺诈、金融欺诈、证据收集
- **知识产权保护**：侵权监测、维权取证、侵权方追踪
- **网络诈骗追踪**：诈骗溯源、资金追踪、团伙画像
- **艺术品鉴定**：真伪验证、来源追溯、著录查询

**触发条件**：当用户提及以下场景时使用
- "帮我调查一下这家公司"
- "分析一下竞争对手"
- "查一下这个人的背景"
- "验证这个信息是否真实"
- "收集XX行业的最新动态"
- "做一份竞品分析报告"
- "追踪这个网络攻击来源"
- "核实这篇新闻报道"
- "调查这起诈骗案件"
- "帮我做背景调查"

## 二、核心方法论（学术基础）

### 2.1 OSINT定义（Coulthart & Nussbaum, 2025）

> "OSINT是**合法获取**的公开或商业信息，经过**验证、分析、传播**后满足特定情报需求。"

**关键区分**：
- **OSINF（Open Source Information）**：原始公开信息，未经加工
- **OSINT**：经过系统化处理后的情报产品
- **OSINV（Open Source Investigation）**：民间调查机构使用的术语（如Bellingcat）

### 2.2 标准情报循环（Intelligence Cycle）

```
规划定向 → 信息采集 → 处理加工 → 分析研判 → 报告传播 → 反馈评估
    ↑                                                              ↓
    ←←←←←←←←←←←←←← 持续迭代优化 ←←←←←←←←←←←←←←←←←←←←←←←←←←←
```

### 2.3 OSINT工作流（IEEE标准）

| 阶段 | 核心任务 | 输出 |
|------|----------|------|
| **1. 规划定向** | 明确目标、范围、合规边界 | 调查计划 |
| **2. 信息采集** | 多源搜集、工具辅助 | 原始数据集 |
| **3. 处理加工** | 清洗、去重、结构化 | 可分析数据 |
| **4. 分析研判** | 交叉验证、假设检验 | 情报产品 |
| **5. 报告传播** | 可视化、决策建议 | 情报报告 |
| **6. 反馈优化** | 评估效用、改进流程 | 方法迭代 |

## 三、信源评估体系

### 3.1 R2C2筛选原则

对每条信息执行四重验证：

| 维度 | 问题 | 判断标准 |
|------|------|----------|
| **R**elevant（相关） | 是否与目标相关？ | 时间、地域、主题是否匹配 |
| **R**eliable（可靠） | 来源是否可靠？ | 官方>学术>权威媒体>自媒体 |
| **C**redible（可信） | 内容是否可信？ | 逻辑自洽、无矛盾、合理 |
| **C**orroborated（佐证） | 是否有佐证？ | ≥2个独立来源印证 |

### 3.2 信源分级体系（NATO Admiralty Code）

| 等级 | 信源类型 | 采信程度 | 示例 |
|------|----------|----------|------|
| **A** | 多源交叉验证（专利+官网+学术三方一致） | 直接采信，高置信度 | 宁德时代BMS5.0：专利+官网+媒体报道一致 |
| **B** | 官方单源（官网、财报、专利） | 可采信，标注单源 | 公司年报、专利文件 |
| **C** | 权威媒体（有采编资质） | 需追溯原始来源后采信 | 36氪、财新、行业期刊 |
| **D** | 社交媒体、待确认信息 | 仅作线索，需验证 | 微博、知乎、公众号 |

### 3.3 Three Confirms验证规则

| 确认数量 | 等级 | 说明 | 行动建议 |
|----------|------|------|----------|
| 1个来源 | 好奇（Curiosity） | 仅作为线索追踪 | 继续搜集更多信息 |
| 2个来源 | 前景（Prospect） | 值得深入调查 | 开始系统验证 |
| 3个独立来源 | 工作级事实（Working Fact） | 可纳入情报结论 | 记录并采信 |

**⚠️ 循环引用警告**：
- Reuters引用BBC，BBC引用Reuters = **单一来源循环**
- 多个自媒体转载同一新闻稿 = **单一来源扩散**
- 必须追溯到**原始发布源**才算独立来源

## 四、分析框架

### 4.1 ACH竞争假设分析法（CIA标准方法）

> 由CIA资深分析师Richards J. Heuer Jr.于1970年代开发，用于对抗认知偏差。

**核心理念**：用证据**证伪**假设，而非**证实**假设。

**执行步骤**：

#### Step 1：假设生成
- 列出**所有可能**的假设，包括"不太可能"的
- 使用多元化团队头脑风暴，避免个人偏见
- 不预设"最可能"假设

#### Step 2：证据列表
- 列出所有相关证据和论据
- 包括假设和逻辑推论
- 区分"支持性证据"和"诊断性证据"

#### Step 3：构建假设-证据矩阵

```
           假设1    假设2    假设3    假设4
证据A        +        -        N        +
证据B        -        +        +        -
证据C        N        +        -        N
证据D        +        +        -        -
------------------------------------------
不一致度     2        2        2        2
```

**符号说明**：
- `+` 证据与假设一致
- `-` 证据与假设不一致（**最有价值**）
- `N` 中性/无关

#### Step 4：诊断性分析
- **横向阅读**：对每个证据，检查它对区分假设的作用
- 诊断性证据：能有效区分不同假设的证据
- 非诊断性证据：对所有假设都一致或不一致

#### Step 5：初步结论
- **不一致度最低**的假设最可能成立
- 不强求唯一答案，可保留多个候选假设
- 用敏感度分析测试结论稳健性

#### Step 6：敏感度分析
- 问：如果关键证据是错误的/误导的，结论会如何变化？
- 问：如果缺失某个关键信息，结论会如何变化？
- 识别结论的"关键支撑点"和"关键假设"

### 4.2 反例校验步骤（新增）

> 基于社区反馈：先写结论，再主动找会让结论失效的场景。

**执行方法**：

1. **写结论时**：同步标注"在什么情况下这条结论会失效"
   - 示例：结论"宁德时代SOC精度达±3%" → 失效条件"仅适用于磷酸铁锂电池、常温环境"

2. **主动寻找反例**：
   - 搜索"XX失败"、"XX问题"、"XX缺陷"
   - 检查是否有相反的专利/论文/报道
   - 如果找到反例，评估其权重

3. **记录失效边界**：
   - 每条结论附带"置信度"和"适用边界"
   - 不确定的内容明确标注"待进一步验证"

### 4.3 踩坑清单（新增）

> 记录失败案例比记录成功案例更有价值。

**常见踩坑类型**：

| 类型 | 示例 | 预防措施 |
|------|------|----------|
| **信源循环** | 多个媒体转载同一通稿 | 追溯原始发布源 |
| **时间错位** | 引用过期专利/旧闻 | 检查发布时间 |
| **过度解读** | 从单一数据点推导趋势 | 要求多时间点验证 |
| **认知偏差** | 只找支持结论的证据 | 使用ACH强制证伪 |
| **语言误译** | 机器翻译歧义 | 核对原文术语 |
| **付费内容** | 引用付费报告摘要 | 标注"基于公开摘要" |

## 五、执行流程

### Step 1：需求澄清（必须）

在开始调查前，必须明确以下要素：

1. **情报目标**：要回答什么具体问题？
2. **调查范围**：对象是谁？时间范围？地域范围？
3. **交付形式**：情报简报？详细报告？结构化数据？
4. **优先级**：核心问题 vs nice-to-have信息
5. **合规边界**：是否涉及敏感信息？法律风险？

**执行方式**：向用户提问确认需求，例如：
- "您想了解该公司的哪些方面？（技术/产品/市场/团队/财务）"
- "调查的时间范围是多久？（近一年/近三年/全周期）"
- "报告的受众是谁？（内部决策层/投资方/法务）"

### Step 2：信息搜集

#### 2.1 信源优先级排序（从高到低）

| 优先级 | 信源类型 | 特点 | 示例平台 |
|--------|----------|------|----------|
| 1 | 专利文献 | 技术细节、法律效力 | CNIPA、USPTO、EPO、WIPO |
| 2 | 官方发布 | 权威性高、时效性好 | 厂商官网、财报、白皮书 |
| 3 | 学术来源 | 深度分析、同行评审 | IEEE、Elsevier、CNKI、arXiv |
| 4 | 权威媒体 | 信息整合、行业洞察 | 36氪、财新、行业期刊 |
| 5 | 公共记录 | 法律效力、公开透明 | 工商注册、法院文书、招投标 |
| 6 | 社交媒体 | 线索源、舆情监测 | 微博、公众号、论坛 |

#### 2.2 搜索策略

**Google Dorking 高级语法**：
```
# 文件类型搜索
filetype:pdf "技术白皮书"
filetype:xls "财务数据"

# 站点限定
site:linkedin.com/in/ "工程师"
site:github.com "source code"
site:cnipa.gov.cn "专利号"

# 时间范围
after:2024-01-01
before:2024-12-31

# 组合查询
site:.gov filetype:xls "tender"
intitle:"年度报告" filetype:pdf
```

**专业搜索引擎**：
- Shodan：联网设备搜索引擎
- Hunter：邮箱地址搜索
- ZoomEye/FOFA：中国网络空间测绘

### Step 3：信息处理

对收集的信息进行结构化处理：

1. **清洗**：去除无关信息、噪音广告
2. **去重**：合并重复内容，识别循环引用
3. **提取**：识别关键实体（人名、公司名、时间、地点、数据）
4. **翻译**：处理外语内容，核对专业术语
5. **分级**：按信源等级分类存储
6. **归档**：保留原始链接和截图，便于追溯

### Step 4：分析研判

#### 4.1 交叉验证
- 检查多个独立来源是否一致
- 识别循环引用（伪多源）
- 评估证据链完整性
- 使用ACH矩阵进行假设检验

#### 4.2 关联分析
- 建立时间线
- 绘制关系图谱（人-公司-事件）
- 发现隐藏关联
- 识别关键节点

#### 4.3 反例校验
- 主动搜索相反证据
- 标注结论的失效边界
- 记录不确定因素

### Step 5：报告输出

#### 输出结构要求

## 六、伦理与合规

### 6.1 禁止事项

❌ **禁止使用的信源**：
- 今日头条、百家号、无来源自媒体
- 知乎个人专栏（中国人物调查）
- 百度百科/知道（中国人物调查）
- 无依据传言、匿名爆料

❌ **禁止行为**：
- 使用非法手段获取信息
- 侵犯个人隐私
- 进行社会工程攻击
- 假冒身份获取信息

### 6.2 合规要求

✅ **必须遵守**：
- 只收集**公开可获取**的信息
- 遵守当地法律法规
- 尊重数据隐私规定（GDPR、个保法）
- 标注信息来源和置信度
- 明确区分"事实"与"判断"

### 6.3 质量控制

| 检查项 | 要求 |
|--------|------|
| 信源追溯 | 每条信息必须有可追溯的原始来源 |
| 多源验证 | 关键结论需≥2个独立来源 |
| 时效检查 | 标注信息获取时间，识别过期信息 |
| 偏差自检 | 使用ACH方法对抗确认偏差 |

## 七、实战案例（新增）

> 基于社区反馈：通过真实案例展示OSINT方法论的落地应用。

### 7.1 案例1：友商电池算法技术追踪

**情报目标**：追踪宁德时代最新BMS算法技术进展

**执行过程**：

#### Step 1：需求澄清
- 调查对象：宁德时代
- 关注维度：SOC/SOH算法、均衡技术、AI应用
- 时间范围：近6个月
- 交付形式：结构化情报报告

#### Step 2：多源搜集
| 信源 | 发现内容 | 信源等级 |
|------|----------|----------|
| CNIPA专利 | "混合电芯SOC估算方法"专利公开 | B |
| 官网新闻 | BMS5.0产品发布会预告 | B |
| 学术论文 | IEEE发表磷酸铁锂SOC修正论文 | B |
| 36氪报道 | 产品发布会现场报道 | C |

#### Step 3：R2C2验证
- **Relevant**：全部与BMS算法相关 ✅
- **Reliable**：专利+官网+学术三方独立 ✅
- **Credible**：数据一致，无矛盾 ✅
- **Corroborated**：多源交叉验证一致 ✅

#### Step 4：ACH假设检验

**假设矩阵**：
```
           假设1      假设2      假设3
          (传统方法)  (AI混合)   (纯AI)
专利描述     +          +          -
官网宣称     -          +          -
论文数据     -          +          -
媒体报道     N          +          N
-----------------------------------
不一致度     2          0          2
```

**结论**：假设2（AI混合架构）不一致度最低，采信。

#### Step 5：反例校验
- 主动搜索"宁德时代SOC误差"：无负面报道
- 查询竞品对比报告：精度处于行业第一梯队
- **失效条件**：仅适用于磷酸铁锂电池、常温环境

#### Step 6：报告输出

**核心结论**：
- 宁德时代BMS5.0采用"卡尔曼滤波+深度学习"混合架构
- 磷酸铁锂SOC估算精度达±3%（行业首次）
- 信源等级：A级（三方一致）
- 置信度：高

### 7.4 常见踩坑实录

#### 踩坑案例1：循环引用陷阱

**错误场景**：
- 发现A媒体引用B媒体报道
- B媒体实际转载自C媒体通稿
- C媒体引用的是企业新闻稿
- **实际来源**：单一企业发布，被误判为"多源验证"

**正确做法**：
```
追溯链条：A→B→C→企业官网
实际独立来源：仅1个（企业官网）
```

#### 踩坑案例2：时间错位

**错误场景**：
- 引用2024年专利数据
- 标注为"最新技术进展"
- 实际该专利已于2025年被新专利替代

**正确做法**：
- 检查专利法律状态
- 搜索后续改进专利
- 标注"截至XX日期"

#### 踩坑案例3：翻译歧义

**错误场景**：
- 英文论文"efficiency improved by 35%"
- 误译为"效率提升35%"
- 实际应为"效率提升了35个百分点"或"效率是原来的1.35倍"

**正确做法**：
- 核对原文数据
- 确认"improved by"vs"improved to"
- 标注原始数据避免歧义
